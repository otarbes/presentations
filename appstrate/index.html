Plateforme open-source d'agent runtime
01 — Vue d'ensemble

Appstrate est la couche plateforme qui manque pour passer des agents IA autonomes à l'échelle d'une équipe ou d'une entreprise.

Plateforme open source pour déployer et opérer des automatisations agentiques avec n'importe quel LLM. Avec sécurité, souveraineté, gouvernance, et le choix du modèle.

État des lieux et avantages

Deux familles de solutions existent déjà. Aucune ne couvre ce qu'il faut pour passer à l'échelle avec sécurité et souveraineté. Deux angles de comparaison, selon le profil de l'évaluateur.

Les suites IA (Claude Cowork, ChatGPT Team, Microsoft Copilot, etc.) sont pensées pour un humain qui clique. Dès qu'une entreprise veut industrialiser, déléguer ou auditer, elle tape un mur :

Les runtimes managés (Bedrock AgentCore, Claude Managed Agents, OpenAI Assistants, Vertex Agent Engine, Gemini Managed Agents, LangChain Managed Deep Agents) règlent le problème de l'industrialisation, mais en gardant le control plane (planning, orchestration, contexte) chez eux, même quand ils ouvrent partiellement l'exécution. Appstrate règle le même problème avec le runtime entier open source :

› Histoire de la catégorie née il y a moins d'un an

La vraie catégorie "managed agent runtime" au sens strict (vendor qui gère lui-même l'isolation d'exécution : microVM, sandbox éphémère pour le code de l'agent) est née en juillet 2025 avec AWS Bedrock AgentCore. En moins d'un an, 5 acteurs majeurs ont rejoint la catégorie. Tous closed source. Appstrate est le seul open source du lot.

Avant juillet 2025, OpenAI Assistants (nov 2023), Bedrock Agents (nov 2023) et Vertex Agent Builder (avril 2024) faisaient de l'orchestration agentique managée mais sans sandbox d'exécution propre : function calling = votre code, action groups = vos Lambdas. C'étaient des précurseurs, pas des full runtimes au sens où AgentCore l'a redéfini.

À qui ça parle

Entreprises et scale-ups

Vous voulez que vos équipes utilisent l'IA en production sans confier les données et les credentials à un vendor cloud. Vous cherchez gouvernance, audit, contrôle des coûts.

Plateformes et éditeurs

Vous voulez livrer des automatisations agentiques à vos clients sans construire la couche plateforme vous-même (sandbox, vault, RBAC, audit). Vous vous concentrez sur vos agents métier.

02 — Sécurité et souveraineté

Vos données ne quittent jamais votre périmètre.

Appstrate est conçue pour les environnements sensibles : secteur public, santé, défense, finance. Cinq piliers rendent la plateforme déployable même dans un cadre air-gap.

BYOK et multi-abonnement

Vos propres clés API, plusieurs comptes (votre abo Claude pour le quota standard, votre abo OpenAI pour le débordement, votre endpoint on-prem pour le sensible). Les coûts LLM restent dans votre relation avec le vendor.

Fallback multi-provider

Quand un quota Anthropic est atteint, un agent bascule sur OpenAI. Quand un modèle baisse en qualité, vous changez sans toucher au code. Le runtime fait l'arbitrage.

Bon modèle, bonne tâche

Agents de masse sur Qwen, agents premium sur Claude, agents sensibles sur Llama. Configurable par workspace ou par agent, sans dupliquer la logique.

Pas de pari technologique

Le marché des LLM bouge tous les trimestres. Vous pariez sur votre capacité à orchestrer des agents, pas sur le vainqueur de la course aux modèles.

Audit et refacturation

Toute action dans la plateforme produit un événement audit. Les coûts LLM et les runs sont rattachés à un workspace, un end-user, un run. Trail exportable au format OpenTelemetry vers Datadog, Grafana, ou votre SIEM.

Refacturation interne immédiate

Coûts LLM et runs comptabilisés par workspace, par end-user, par run. Vous savez exactement combien le département RH ou un client final consomme. Pas de calcul a posteriori.

Impersonation auditée

Un agent peut agir au nom d'un user spécifique (avec son OAuth, ses droits). Chaque action est tracée par utilisateur et par run. Compliance prête.

OpenTelemetry standard

Traces exportables vers votre stack d'observabilité. Pas de format propriétaire. Pas de secret dans les traces.

Vue globale et vue locale

Le DSI a la vue globale. Le responsable d'un workspace gère ses agents et ses users sans toucher au reste. Pas de tout-ou-rien sur les permissions.

Pattern 01
Code dans les tools, pas dans le prompt

Toute la plomberie (list Fathom, fetch transcripts, push GitHub) est dans 2 tools custom. Le LLM orchestre uniquement la séquence, pas un curl ni un bash.

Pattern 02
Idempotent par design

Un état persistant pin("sync_state") contient les IDs déjà traités. Les runs successifs ne retraitent jamais ce qui est déjà fait. Sécurité contre les doublons.

Pattern 03
Streaming hors contexte LLM

Les transcripts (parfois lourds) sont écrits dans /tmp/, jamais dans le contexte du modèle. Évite de saturer les tokens, lecture par le tool en local.

Pattern 04
Multi-providers parallèles

Un seul tool call gère les 4 comptes Fathom en parallèle. Le LLM ne séquence pas manuellement. Latence divisée par 4.

Pattern 01
Chaîne d'agents découplés

Deux agents indépendants qui communiquent par le filesystem Drive (et un factures-parsed.json). Chacun peut être redéployé, débugué, replacé sans toucher à l'autre.

Pattern 02
Skill métier partagé

Un seul skill compta-references (regex BNC, vendor-rules, fiscal-year) consommé par les deux agents. Single source of truth, pas de duplication des règles métier.

Pattern 03
LLM raisonne, code calcule

La classification des PDFs (facture vs dépôt vs Stripe vs gouv) est faite par le LLM. La conciliation factures↔transactions (matching financier) est faite par un script déterministe. Chacun fait ce qu'il fait le mieux.

Pattern 04
Multimodal en fallback

Si un PDF est scanné (pas de texte extractible), le LLM bascule automatiquement en lecture image. Pas d'OCR externe à maintenir, pas de service tiers dans la boucle.

Pattern 01
Portage zéro réécriture

Le skill Claude Code existant (SKILL.md, scripts Python, xlsx gabarit, références éditoriales) a été reconditionné en .afps et importé tel quel. Seuls les appels MCP/HTTP locaux deviennent des provider_call sandbox-isolés. Investir dans des skills Claude Code prépare déjà la migration vers Appstrate.

Pattern 02
Prompt agent minimaliste (22 lignes)

Le savoir métier est dans le skill, pas dans le prompt. L'agent ne fait que charger le skill, choisir le workflow et déléguer. Skill et agent versionnés indépendamment, skill réutilisable dans plusieurs agents.

Pattern 03
Connecteur custom en quelques minutes

Directus n'est pas dans les 60+ providers built-in. Le provider @tastet/directus-staging = 1 manifest + 1 PROVIDER.md (40 lignes), packagé en .afps, importé en une commande. Pattern reproductible pour tout SaaS interne (CMS, CRM, ERP).

Pattern 04
Sandbox Linux complet, pas une cage JS

Python + pip install openpyxl tournent dans le sandbox sans portage TS. Les scripts du skill original (manipulation xlsx, parsing) s'exécutent tels quels. Le sandbox est un vrai environnement Linux.

À noter : ces trois agents sont packagés au format AFPS (manifest + prompt + dépendances), versionnés, et redéployables sur n'importe quelle instance Appstrate. Ils ont été développés ou portés en quelques heures chacun grâce au skill Claude Code.
